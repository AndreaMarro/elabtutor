Git as Memory\n\nFrom the [DEV Community article on git log as memory](https://dev.to/0coceo/git-log-as-memory-how-an-ai-ceo-maintains-continuity-across-session-boundaries-1cc0): commit history is timestamped, immutable, and more reliable than human memory.\n\n**Practical extraction for a new session:**\n\n```bash\n# Last 20 commits with stats (what changed recently)\ngit log --oneline --stat -20\n\n# Changes since last session tag\ngit log --oneline session-115..HEAD\n\n# Files changed in last 7 days\ngit log --since=\"7 days ago\" --name-only --pretty=format: | sort -u\n\n# Summary of recent work for AI context injection\ngit log --since=\"3 days ago\" --pretty=format:\"- %s (%ar)\" | head -20\n```\n\n**Session tagging pattern:**\n\n```bash\n# At session end\ngit tag \"session-116\" -m \"LCD blocks, ScratchEditor chunk investigation\"\n\n# At session start, see what happened\ngit log session-115..session-116 --oneline\n```\n\n**Automated context injection** (add to `.claude/hooks/` or a startup script):\n\n```bash\n#!/bin/bash\n# .claude/scripts/git-context.sh\necho \"## Recent Git Activity (last 7 days)\"\ngit log --since=\"7 days ago\" --pretty=format:\"- **%s** (%ar by %an)\" | head -15\necho \"\"\necho \"## Files Most Changed Recently\"\ngit log --since=\"14 days ago\" --name-only --pretty=format: | sort | uniq -c | sort -rn | head -10\n```\n\nThe [ctx CLI tool](https://dev.to/lakshmisravyavedantham/i-built-a-cli-that-gives-any-ai-instant-context-about-your-project-3ig8) automates this: generates a context pack with structure map, last 10 commits, README summary, and custom notes — one Markdown file any AI can parse.\n\n---\n\n## 5. Hierarchical Memory Architecture (L0/L1/L2)\n\nThis is the most actionable section. Here is a concrete implementation using current Claude Code tools:\n\n```\n.claude/projects/{project}/memory/\n├── MEMORY.md                    # L0: Always loaded (first 200 lines)\n├── state.json                   # L0: Machine-readable state\n├── handoff.md                   # L0: Last session's handoff\n├── architecture.md              # L1: Loaded on demand\n├── decisions.md                 # L1: Loaded on demand\n├── scores.md                    # L1: Loaded on demand\n├── known-issues.md              # L1: Loaded on demand\n├── domain/\n│   ├── scratch-blockly.md       # L1: Domain-specific\n│   ├── galileo-ai.md            # L1: Domain-specific\n│   ├── vision-system.md         # L1: Domain-specific\n│   └── ipad-responsive.md       # L1: Domain-specific\n├── archive/\n│   ├── sessions-1-30.md         # L2: Deep archive\n│   ├── sessions-31-60.md        # L2: Deep archive\n│   ├── sessions-61-90.md        # L2: Deep archive\n│   └── sessions-91-115.md       # L2: Deep archive\n└── vectors/                     # L2: ChromaDB (optional)\n    └── chroma.sqlite3\n```\n\n**Loading rules (put in CLAUDE.md):**\n\n```markdown\n## Memory Loading Protocol\n- L0 (MEMORY.md + handoff.md): Auto-loaded every session. Keep under 200 lines total.\n- L1 (topic files): Read ONLY when the current task relates to that domain.\n  Example: working on iPad layout → read memory/domain/ipad-responsive.md\n- L2 (archive/): Read ONLY when explicitly asked about old sessions or when\n  debugging a regression that might relate to past work.\n- NEVER load all L1+L2 files at once. Budget: max 3 topic files per session.\n```\n\n**Why this works:** Claude Code already supports this pattern natively. MEMORY.md is loaded automatically (200 lines). Topic files are read on demand using standard file tools. The key is making the index good enough that Claude knows WHICH file to read.\n\n---\n\n## 7. Context Compression Techniques\n\nFrom the research on [ACON](https://arxiv.org/pdf/2510.00615), [Focus](https://arxiv.org/html/2601.07190), and [Factory.ai](https://factory.ai/news/compressing-context):\n\n**Practical for your case — incremental summarization:**\n\nInstead of keeping all 115 sessions in detail, compress in waves:\n\n```\nSessions 1-30:   → 1 paragraph summary (what was built)\nSessions 31-60:  → 5-10 bullet points (key milestones)\nSessions 61-90:  → 15-20 bullet points (recent context)\nSessions 91-115: → Full handoff notes (active context)\n```\n\n**The \"sawtooth\" pattern** from Focus architecture: context grows during active work and collapses during consolidation. At session end, consolidate key learnings into persistent files and prune raw interaction history.\n\n**Concrete compression script** (run every 10 sessions):\n\n```bash\n# archive-sessions.sh\n# Move sessions older than 30 into compressed archive\n# Keep last 30 sessions in full detail\n# Keep last 10 as individual handoffs\n# Older ones get merged into decade summaries\n```\n\n**LLMLingua** ([llmlingua.com](https://llmlingua.com/)) achieves 20x compression with <2% performance loss, but it requires a separate model to run the compression. For your use case, manual summarization by Claude at session end is more practical — just add it to the session protocol.\n\n---\n\n## 8.

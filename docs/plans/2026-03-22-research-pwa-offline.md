AI Fallback Offline: Practical Options for a Tutor\n\nBased on research, there are three tiers of offline AI, from simplest to most ambitious:\n\n### Tier 1: Pre-Computed Response Cache (Recommended First Step)\n\nCache the most common Galileo responses. IndexedDB vs SQLite-WASM for Student Data\n\n### Recommendation: **IndexedDB via Dexie.js** (or RxDB for sync)\n\nFor ELAB's student data (progress, circuit saves, quiz results), **IndexedDB is the right choice**. Here's why:\n\n| Factor | IndexedDB (via Dexie.js) | SQLite-WASM |\n|--------|-------------------------|-------------|\n| Setup overhead | Zero -- native browser API | ~500ms init + 3MB WASM download |\n| Persistence | Native | Requires OPFS (Worker-only) or serialize to IndexedDB |\n| Browser support | Universal | OPFS only in Chromium; Safari partial |\n| Complex queries | Limited (index-based) | Full SQL (JOINs, subqueries) |\n| Data complexity | ELAB needs: key-value, simple lists | Overkill for student progress |\n| iPad support | Full | OPFS: needs Web Worker workaround |\n\n**ELAB's data is not complex enough for SQL.** Student data is:\n- Progress per experiment (experiment_id, completed_steps, score, timestamp)\n- Circuit saves (experiment_id, serialized_state_json)\n- Quiz results (quiz_id, answers, score, timestamp)\n- Chat history (session_id, messages_array)\n\nThis is straightforward key-value/document storage. IndexedDB handles it natively.\n\n### Dexie.js Implementation\n\n[Dexie.js](https://dexie.org/) is the recommended wrapper -- it provides a clean Promise-based API over IndexedDB:\n\n```js\n// src/db/elabDB.js\nimport Dexie from 'dexie';\n\nexport const db = new Dexie('ElabTutorDB');\n\ndb.version(1).stores({\n  progress: '++id, experimentId, studentId, timestamp',\n  circuits: '++id, experimentId, studentId, [experimentId+studentId]',\n  quizResults: '++id, quizId, studentId, timestamp',\n  chatHistory: '++id, sessionId, timestamp',\n  pendingSync: '++id, type, payload, createdAt', // Queue for offline changes\n});\n\n// Save progress offline\nexport async function saveProgress(experimentId, data) {\n  await db.progress.put({\n    experimentId,\n    studentId: getCurrentStudentId(),\n    ...data,\n    timestamp: Date.now(),\n    synced: false,\n  });\n  // Queue for sync\n  await db.pendingSync.add({\n    type: 'progress',\n    payload: { experimentId, ...data },\n    createdAt: Date.now(),\n  });\n}\n```\n\n### Why NOT SQLite-WASM\n\nA [real-world comparison](https://haroonwaves.com/blog/building-email-client) found that migrating to SQLite-WASM cut load times by 100x -- but that was for an email client with complex relational queries across thousands of records. ELAB has ~67 experiments with simple per-student records. The architectural overhead of OPFS + Web Workers + WASM initialization is not justified.\n\n**One caveat from [RxDB docs](https://rxdb.info/articles/offline-database.html)**: Apple Safari deletes IndexedDB data after 7 days of non-use for non-installed PWAs. This is a strong reason to encourage PWA installation (see section 6).\n\n---\n\n## 5. Sync Strategies: Offline-to-Cloud Reconciliation\n\n### Recommended: Last-Write-Wins + Sync Queue\n\nFor ELAB student data, **Last-Write-Wins (LWW) with a sync queue** is the right approach. CRDTs are overkill because:\n\n- Students don't collaboratively edit the same data simultaneously.\n- Each student has their own progress/quiz data.\n- The only conflict scenario is: student uses ELAB on two devices offline, then both sync. LWW by timestamp is acceptable here (latest progress wins).\n\n### Implementation: Background Sync Queue\n\n```js\n// src/db/syncManager.js\nimport { db } from './elabDB';\n\nexport async function syncPendingChanges() {\n  if (!navigator.onLine) return;\n\n  const pending = await db.pendingSync.toArray();\n  if (pending.length === 0) return;\n\n  try {\n    const response = await fetch('/api/sync', {\n      method: 'POST',\n      headers: { 'Content-Type': 'application/json' },\n      body: JSON.stringify({\n        changes: pending,\n        clientTimestamp: Date.now(),\n      }),\n    });\n\n    if (response.ok) {\n      const { serverChanges } = await response.json();\n      // Apply server-side changes locally (server wins for conflicts)\n      for (const change of serverChanges) {\n        await applyServerChange(change);\n      }\n      // Clear synced items\n      await db.pendingSync.where('id').anyOf(pending.map(p => p.id)).delete();\n    }\n  } catch (err) {\n    console.warn('Sync failed, will retry:', err);\n  }\n}\n\n// Trigger sync on reconnection\nwindow.addEventListener('online', syncPendingChanges);\n\n// Also try periodic sync\nif ('serviceWorker' in navigator && 'SyncManager' in window) {\n  navigator.serviceWorker.ready.then(reg => {\n    reg.sync.register('elab-sync');\n  });\n}\n```\n\nIn the service worker:\n```js\nself.addEventListener('sync', (event) => {\n  if (event.tag === 'elab-sync') {\n    event.waitUntil(syncPendingChanges());\n  }\n});\n```\n\n### Why NOT CRDTs\n\n[As noted by experts](https://www.sachith.co.uk/offline-sync-conflict-resolution-patterns-architecture-trade%E2%80%91offs-practical-guide-feb-19-2026/), CRDTs resolve data structure conflicts but not business logic conflicts. For ELAB:\n- Student progress is append-only (you don't \"undo\" completing an experiment).\n- Quiz results are immutable once submitted.\n- Circuit saves use simple versioning (save slot 1, 2, 3).\n\nLWW with per-student isolation is sufficient. If you ever need collaborative features (teacher + student editing the same circuit), consider [RxDB](https://rxdb.info/replication.html) which provides built-in conflict resolution with replication to any backend.\n\n---\n\n## 6.

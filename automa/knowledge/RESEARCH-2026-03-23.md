# Ricerche del 2026-03-23 — Risultati consolidati

## 1. Karpathy Autoresearch — Pattern keep-or-revert

**Cosa è**: Un loop autonomo che modifica codice, esegue test (5 min), misura una metrica (val_bpb), e decide: keep (commit git) o revert (git reset).

**Come funziona**:
- 630 righe Python, 1 file, 1 GPU, 1 metrica
- ~12 esperimenti/ora, ~100 esperimenti in una notte
- Git è la memoria persistente: commit history = cosa ha funzionato
- L'agent legge la storia dei commit per informare le ipotesi future

**Risultati reali**:
- Karpathy: 89 esperimenti, 15 miglioramenti, 74 scartati, 0 crash
- Shopify CEO: 120 esperimenti, 93 commit, 53% rendering più veloce, tutti i 974 test passati

**Cosa ELAB copia**: Il pattern keep-or-revert con evaluate.py come metrica. Già implementato parzialmente.

**Rischio**: Dopo ~100 esperimenti, i miglioramenti diventano marginali. L'agent ottimizza la metrica esposta — se la metrica è sbagliata, trova scorciatoie. Per questo evaluate.py v2 è più severo.

**Fonte**: [karpathy/autoresearch](https://github.com/karpathy/autoresearch)

## 2. Graphiti vs Mem0 — Memoria per agenti

**Mem0** (Apache 2.0, 48K stars):
- Dual-store: vector DB + knowledge graph opzionale
- Semplice da self-hostare
- Comprime chat in rappresentazioni ottimizzate (80% token reduction)
- Limiti: "context blindness" — non distingue bene entità diverse, top-k crowding
- Graph features solo nel piano Pro ($249/mese)

**Graphiti/Zep**:
- Knowledge graph temporale — il tempo è dimensione nativa
- Traccia QUANDO le relazioni erano vere e come cambiano
- Benchmark LongMemEval: Zep 63.8% vs Mem0 49.0%
- Limiti: richiede Neo4j, operativamente pesante, Community Edition deprecata
- Costo token alto in fase di costruzione grafo (600K+ token/conversazione)

**Decisione per ELAB**: Per ora, SQLite enhanced (già funzionante) + struttura a 4 strati:
- Working memory: task corrente, file rilevanti
- Episodic memory: esperimenti fatti, score prima/dopo
- Semantic memory: regole prodotto, pattern pedagogici
- Graph memory: futuro, quando serve davvero

**Rischio**: Overengineering. SQLite con search_knowledge() funziona già. Aggiungere Graphiti/Mem0 ha senso solo quando la knowledge supera 100+ entries e serve recall preciso.

**Fonti**: [Graphiti vs Mem0 benchmark](https://dev.to/juandastic/i-benchmarked-graphiti-vs-mem0-the-hidden-cost-of-context-blindness-in-ai-memory-4le3), [Cognee evaluation](https://www.cognee.ai/blog/deep-dives/ai-memory-tools-evaluation)

## 3. Web Speech API — Voce per bambini in italiano

**Supporto browser**: Chrome/Edge supportano Speech Recognition. Safari supporto parziale. Firefox NO.

**Italiano**: Supportato con voce `it-IT`. Microsoft Isabella (Natural) disponibile su Edge.

**Per bambini**: Nessun framework specifico trovato. La Web Speech API riconosce bene adulti ma può avere difficoltà con voci acute di bambini e pronuncia non standard.

**Decisione per ELAB**: Prototipo con Web Speech API (gratuito, browser-native) per comandi semplici al simulatore ("metti un LED", "avvia la simulazione"). NON per dettatura lunga.

**Pipeline target**: utente → Web Speech API (ASR) → intent parser → DSL comandi → validator → simulatore

**Rischio ALTO**: Bambini parlano in modo imprevedibile. Il riconoscimento vocale potrebbe frustrare più che aiutare. Serve SEMPRE un fallback testuale. Il validator è CRITICO — senza validator, comandi sbagliati vanno al simulatore.

**Fonti**: [Web Speech API spec](https://webaudio.github.io/web-speech-api/), [Can I Use](https://caniuse.com/speech-recognition)

## 4. Latenza Galileo — Come ridurla da 17.6s a <5s

**Analisi del collo di bottiglia** (ipotesi — da verificare con breakdown):
- Browser → Nanobot (Render free tier): cold start fino a 30s se il container dorme
- Nanobot → n8n (Hostinger): network hop aggiuntivo
- n8n → Anthropic API: tempo di generazione LLM

**Tecniche di riduzione**:
1. **Warm-up**: ping periodico al nanobot ogni 5 min per evitare cold start Render
2. **Streaming**: mostrare token per token (percepito più veloce)
3. **Prompt più corto**: ridurre system prompt = meno input tokens = risposta più veloce
4. **Cache**: risposte pre-calcolate per domande frequenti ("cos'è un LED?")
5. **VPS dedicato**: spostare nanobot dal free tier Render al VPS (se budget permette)

**Decisione per ELAB**: Prima misurare il breakdown reale (dove vanno i 17.6s). Poi applicare warm-up (facile, gratis) e cache (medio). Streaming è un cambio UI significativo.

## 5. GDPR per piattaforme educative minori in Italia

**Requisiti chiave**:
- Sotto 14 anni: serve consenso del genitore/tutore (art. 8 GDPR, art. 2-quinquies Codice Privacy italiano)
- Data minimization obbligatoria
- Privacy by design e by default
- No profilazione automatizzata su minori
- Informativa privacy in linguaggio comprensibile per bambini

**Per ELAB**: Non raccogliere dati personali se possibile. Se serve analytics, pseudonimizzare. Log di chat con Galileo: NON salvare dati identificabili. Cookie banner con opt-out di default.

**Rischio**: Se ELAB salva conversazioni con Galileo che contengono nomi/scuole/dati personali → violazione GDPR. Serve audit dei dati raccolti.

## 6. Misconcezioni elettricità bambini 8-14 (da letteratura)

**Top 3 misconcezioni documentate**:
1. **Modello unipolare**: i bambini pensano che la corrente "esce" dalla batteria da un solo polo e "arriva" al componente. Non capiscono il circuito chiuso.
2. **Corrente si consuma**: credono che la corrente "si usi" lungo il circuito, quindi dopo il LED ne resta meno.
3. **Batteria = serbatoio**: vedono la batteria come un contenitore di corrente che si svuota, non come una pompa che mantiene una differenza di potenziale.

**Per Galileo**: Queste misconcezioni sono PREVEDIBILI. Galileo dovrebbe:
- Riconoscere frasi che indicano queste misconcezioni
- Correggere con analogie appropriate (acqua nel tubo per il circuito chiuso)
- NON usare terminologia che rinforza le misconcezioni

**Fonte**: Letteratura pedagogica consolidata (Shipstone 1984, Osborne 1983, Driver & Squires 1985)
